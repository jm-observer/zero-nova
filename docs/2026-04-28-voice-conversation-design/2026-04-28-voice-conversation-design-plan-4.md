## 详细设计

### 1. 测试分层
- 前端单元测试：
  - `voice.ts` 状态机
  - `voice-overlay.ts` UI 状态映射
  - `gateway-messages.ts` 语音协议解析
- 协议契约测试：
  - `nova-protocol` 新增语音消息 schema/serde 往返测试
- 后端单元/集成测试：
  - 语音编排层在 STT、聊天、TTS 三段成功/失败时的行为
  - 使用 `MockSttProvider` / `MockTtsProvider`（Plan 3 中定义的 mock 供应商实现）隔离外部依赖
  - 测试并发拒绝逻辑（第二个 transcribe 请求在前一个处理中时应返回 `voice_request_in_progress`）
- 端到端测试：
  - 用固定夹具音频（16kHz mono WAV）验证识别、问答和朗读流程

### 1.1 性能基准
- 建议为 MVP 设定以下延迟基准（端到端，不含网络传输）：
  - **STT 延迟**：从音频上传完成到转写文本返回，P95 < 3s（5 秒以内的语音输入）
  - **TTS 延迟**：从文本提交到音频返回，P95 < 2s（100 字以内的文本）
  - **全链路延迟**：从录音结束到助手回复文本显示，P95 < 10s（含 LLM 响应时间）
- 基准值仅作为首版参考指标，具体取决于供应商选型；建议在端到端测试中记录实际耗时并输出报告

### 2. Mock 供应商策略
- mock 实现位于 `crates/nova-agent/src/voice/mock.rs`（Plan 3 新增）。
- `MockSttProvider`：接收任意音频，返回可配置的固定文本（默认 `”这是一段测试语音”`）和固定置信度。
- `MockTtsProvider`：接收任意文本，返回一段固定的静音 MP3 音频（内嵌在代码中或从 fixtures 加载）。
- 供应商选择通过 `VoiceConfig` 控制，建议新增 `stt_provider` / `tts_provider` 字段：
  - `”mock”` — 使用 mock 供应商
  - `”openai_compat”` — 使用 OpenAI 兼容供应商（默认值）
- 开发环境和 CI 中默认使用 mock，无需配置真实 API key 即可运行语音相关测试。

### 3. 首版灰度策略
- 推荐按以下顺序开放：
  - 开发环境 + mock STT/TTS（验证全链路数据流通）
  - 内部测试账号 + 真正供应商（验证识别准确率和合成质量）
  - 小范围开启”语音输入”（仅 STT 链路）
  - 最后再开启”自动朗读回复”和”连续语音模式”
- 这样可以先验证输入链路，再验证输出链路，降低问题排查难度。

### 4. 降级策略
- STT 不可用：
  - 前端保留录音入口但提示“暂不可识别”，或直接隐藏语音输入按钮。
- TTS 不可用：
  - 仍然返回文本回复，只关闭自动朗读。
- 语音流式能力不可用：
  - 自动降级到整段上传模式。

### 5. 推荐实施顺序
- 第 1 步：打通 `录音 -> STT -> 文本消息发送`。
- 第 2 步：打通 `助手文本回复 -> TTS 播放`。
- 第 3 步：补齐 `Barge-In`、连续语音模式和调试观测。
- 第 4 步：再评估是否值得升级为低延迟流式双工。

### 6. 后续演进方向
- 增量识别：在用户还没说完时就显示 partial transcript。
- 边生成边播报：助手文本分句产出后立刻触发 TTS。
- 声纹/角色能力：按 agent 或场景配置不同 TTS voice。
- 本地优先模式：在桌面端可选本地 STT/TTS，弱网时自动切换。

## 测试案例
- 正常路径：MVP 全链路一次成功，且前端/后端日志可完整追踪 request_id。
- 边界条件：灰度关闭 STT 或 TTS 后，UI 正确降级。
- 异常场景：音频夹具损坏、供应商返回空内容、TTS 音频无法播放。
- 回归场景：语音功能关闭时，产品表现与当前主干一致。
