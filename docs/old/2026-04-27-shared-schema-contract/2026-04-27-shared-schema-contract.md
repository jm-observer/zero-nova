# 2026-04-27 前后端共享 Schema 契约设计

## 时间
- 创建日期：2026-04-27
- 最后更新：2026-04-27

## 项目现状
- 协议结构定义集中在 `crates/nova-protocol`（Rust DTO + serde 序列化）。
- 前端 `deskapp/src/core/types.ts` 维护独立手写类型，并通过注释“对齐后端”保持同步。
- 当前仅有后端序列化测试，缺少“前后端同源 schema”与自动契约校验流程。
- 风险表现为：字段新增/重命名/可选性变更时，前端可能编译通过但运行时失配。

## 整体目标
- 建立单一事实来源（Single Source of Truth）：后端协议模型 -> Schema 工件。
- 前后端消费同一份 Schema，实现“编译期类型约束 + 运行时数据校验”。
- 将 Schema 变更纳入 CI 契约门禁，避免协议漂移进入主干。

## Plan 拆分
- Plan 1：Schema 单一来源定义（协议边界与生成范围）。
  - 依赖：无。
  - 产物：Schema 根类型清单、命名规范、版本策略。
- Plan 2：Schema 导出与工件管理流水线。
  - 依赖：Plan 1。
  - 产物：可重复生成的 Schema 文件、目录结构、变更策略。
- Plan 3：前端消费同一 Schema（类型生成 + 运行时校验）。
  - 依赖：Plan 2。
  - 产物：前端自动生成类型、请求/响应校验封装。
- Plan 4：契约测试与 CI 门禁。
  - 依赖：Plan 3。
  - 产物：后端契约测试、前端契约测试、CI 强制检查。

执行顺序：Plan 1 -> Plan 2 -> Plan 3 -> Plan 4。

## 风险与待定项
- 风险 1：协议类型较多（`agent/chat/session/observability`），一次性全量接入会导致改造面过大。
- 风险 2：枚举/联合类型在跨语言映射时可能出现语义损失。
- 风险 3：Schema 生成与前端构建流程耦合后，若缺少缓存会拖慢本地迭代。
- 待定项 1（需确认）：Schema 生成方案选型。
  - 方案 A（确认使用该方案）：Rust 导出 JSON Schema（推荐，后端为事实来源）。
  - 方案 B：手写 JSON Schema（不推荐，维护成本高）。
- 待定项 2（需确认）：前端运行时校验方案。
  - 方案 A（确认使用该方案）：`ajv`（性能高、生态成熟）。
  - 方案 B：`zod`（开发体验好，但需转换或双维护）。
- 待定项 3（确认语义版本化（v1/v2））：Schema 版本策略采用“随仓库主线”还是“语义版本化（v1/v2）”。
