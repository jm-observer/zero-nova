# Plan 2：测试契约与夹具稳定化

## Plan 编号与标题
- Plan 2：测试契约与夹具稳定化

## 前置依赖
- 依赖 Plan 1 完成基础运行链路

## 本次目标

建立面向 E2E 的稳定测试契约，减少对真实后端、随机 UI 状态和脆弱选择器的依赖，让测试具备可重复执行能力。

可验证标准：
- 测试使用的核心选择器可在源码中稳定找到
- 关键页面状态可通过测试夹具显式构造
- 断言以结构和状态为主，不依赖易变文案

## 详细设计

### 1. 选择器契约收敛

当前 E2E 草案与实际页面存在以下错位：

- 测试使用 `#send-button`，实际页面是 `#send-btn`
- 测试使用 `#confirm-dialog`，实际页面是 `#confirm-modal` 或 `#confirm-dialog-overlay`
- 测试使用 `#new-agent-btn`、`#agent-modal`，当前页面未直接提供同名节点
- 测试依赖 `.validation-error`、`.token-stream`，当前未形成稳定契约

本设计建议采用“测试契约显式化”策略：

方案 A：复用现有 `id`
- 适用于已有且稳定的节点，如 `#message-input`、`#send-btn`、`#session-list`

方案 B：新增 `data-testid`
- 适用于会随视觉样式调整的复杂节点，如消息项、流式 token 区、Agent 行、确认弹窗按钮

推荐优先级：
- 先复用稳定 `id`
- 再为动态节点增加 `data-testid`

这样可以避免：
- 测试绑定样式类名
- DOM 结构微调导致大面积断测

### 2. 页面对象与定位器封装

为降低用例文件中的重复选择器和等待逻辑，建议增加轻量页面对象层：

- `chat-page.ts`
- `sidebar-page.ts`
- `modals-page.ts`

职责：
- 统一暴露定位器
- 统一封装常见动作，如发送消息、新建会话、确认删除
- 统一处理等待逻辑，如等待首屏加载、等待消息渲染、等待模态框展开

这样做的原因：
- 选择器变更时只需改一处
- 测试代码更聚焦业务行为而不是 DOM 细节

### 3. 测试夹具设计

E2E 若直接依赖真实网关、真实模型或真实本地文件系统，失败会高度随机。因此需要引入夹具层。

建议按“最小可控闭环”设计：

#### 3.1 应用启动夹具
- 绕过首次引导或预置引导完成状态
- 预置本地存储中的必要字段，如语言、当前会话、Agent 基础信息

#### 3.2 网关通信夹具
- 拦截前端对网关的调用，返回静态可预期数据
- 聊天响应采用预定义消息片段，而不连真实模型
- 会话列表、Agent 列表使用固定 fixture 数据集

#### 3.3 Tauri API 夹具
- 对 `invoke` 等桌面 API 调用给出测试替身
- 保证 Web 壳层运行时不会因缺失桌面环境直接崩溃

夹具优先级：
- 优先 mock 前端边界
- 尽量不 mock 组件内部逻辑

### 4. 稳定等待策略

当前草案中存在直接等待某个 class 出现的写法，这在异步 UI 中较脆弱。建议统一策略如下：

- 首屏：等待 loading overlay 消失或主容器可交互
- 聊天发送：等待用户消息气泡出现，再等待助手消息容器出现
- 删除确认：等待 modal 去除 `hidden`
- 流式输出：断言“内容长度增长”或“状态标记变化”，而不是依赖具体 token DOM 结构

### 5. 国际化与文案漂移策略

测试不应依赖 `New Chat` 这类可翻译文案作为主断言。

推荐做法：
- 优先断言会话数量增加
- 或断言当前激活会话变化
- 若必须断言标题，优先由夹具预置固定标题，而不是依赖默认文案

## 测试案例

### 正常路径
- 页面加载后，测试可稳定定位输入框、发送按钮、会话列表和 Agent 列表
- 使用固定 fixture 时，消息、会话、Agent 的初始状态可重复复现

### 边界条件
- 在中文和英文环境下，测试主体断言不因文案变化失败
- 在 DOM 结构调整但 `data-testid` 未变的情况下，测试仍可通过

### 异常场景
- 网关返回错误 fixture 时，页面能显示受控错误状态而非卡死
- Tauri API 不可用时，测试环境仍可由 mock 路径启动并进入主界面
