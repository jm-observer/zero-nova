## 本次目标

基于稳定契约和测试夹具，重写当前三组 E2E 用例，使其真正覆盖关键用户路径，并为后续 CI 接入提供清晰边界。

可验证标准：
- 聊天、会话、Agent 三组场景各自至少具备一条稳定 happy path
- 每组场景至少有一条边界或错误路径
- 失败时可输出截图、trace 和可读错误信息

## 详细设计

### 1. 用例重写原则

现有三组脚本存在“场景名正确，但断言与实现脱节”的问题，因此本轮不建议在原脚本上零碎修补，而应按当前页面实现进行重写。

重写原则：
- 一条用例只验证一类核心行为
- 优先验证用户可感知结果，而不是内部实现细节
- 每条用例都基于固定 fixture，避免状态串扰

### 2. 聊天场景设计

#### 2.1 最小 happy path
- 进入页面
- 预置一个可用会话
- 输入消息并点击发送
- 断言用户消息已渲染
- 断言助手响应容器已渲染

#### 2.2 边界路径
- 输入空消息时，发送动作被阻止
- 不强依赖某个错误文案，而是断言按钮状态、输入框校验态或消息数量未增加

#### 2.3 流式响应路径
- 使用 fixture 分片返回助手消息
- 断言助手消息内容逐步增长，或状态从“进行中”转为“完成”

### 3. 会话场景设计

#### 3.1 新建会话
- 记录初始会话数
- 触发新建动作
- 断言会话数加一，且新会话成为激活项

#### 3.2 切换会话
- 预置至少两个会话
- 点击非当前会话
- 断言激活项变化，主聊天区同步切换

#### 3.3 删除会话
- 触发删除动作
- 断言确认模态框显示
- 确认删除后，断言会话数减少或激活项迁移

### 4. Agent 场景设计

当前代码中尚未看到与 `#new-agent-btn` / `#agent-modal` 对应的稳定节点，因此 Agent 用例需要先按真实能力拆分：

#### 4.1 已落地能力
- 列出 Agent
- 切换当前 Agent

#### 4.2 待落地能力
- 新建 Agent

设计要求：
- 若“新建 Agent”尚未形成真实交互入口，本轮不强行保留该用例
- 对未落地场景可先标记为 backlog，而不是写成必失败测试

### 5. 报告与诊断产物

需要保留以下诊断信息：

- HTML report
- failure screenshot
- trace

同时建议在测试失败时附带更可读的步骤描述，例如：
- “等待会话列表完成初始渲染”
- “等待助手消息从 streaming 进入 settled”

### 6. CI 收口策略

本轮设计不强制立即把 E2E 纳入 Rust workspace 的修复循环；原因是当前修复流程面向 `cargo`，而 `deskapp` 属于独立前端工程。

建议分两步推进：

第一步：
- 本地 Windows 稳定通过

第二步：
- 在前端或发布流程中单独增加 E2E job
- 与 Rust `clippy/fmt/test` 并列，而不是混入同一命令

这样能避免：
- 前端浏览器依赖拉长 Rust 检查链路
- 一次前端 flaky 直接阻塞所有本地 Rust 迭代

## 测试案例

### 正常路径
- 聊天发送后，用户消息与助手消息均可见
- 新建会话后，会话数增加且当前会话切换成功
- 切换 Agent 后，列表激活状态更新

### 边界条件
- 空消息发送不产生新消息
- 单会话场景下删除入口或切换行为符合预期
- 仅一个 Agent 时，切换用例自动退化为“当前 Agent 状态保持正确”

### 异常场景
- 网关 fixture 返回失败时，聊天界面显示受控失败状态
- fixture 数据缺失时，测试能快速失败并给出可诊断信息，而不是长时间超时
