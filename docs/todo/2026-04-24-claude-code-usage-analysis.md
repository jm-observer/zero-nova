## 2. 它的提示词组织方式

从导出内容看，这次 Claude Code 采用的是明显的“分层注入”方式，而不是单一系统提示词。

### 2.1 基础系统提示

最外层是 Claude Code 的通用系统提示，核心特点：

- 定义身份：`Claude Code, Anthropic's official CLI`
- 定义任务范围：软件工程协作、代码修改、解释、调试
- 定义安全边界：安全测试允许，但拒绝破坏性攻击、DoS、供应链攻击等
- 定义行为准则：简洁响应、先说明再用工具、避免无关评论、结尾只做简短总结
- 定义工具策略：优先专用工具，复杂任务建议配合 `TaskCreate`

这层提示词比较“产品化”，强调行为一致性和 CLI 场景适配。

### 2.2 延迟工具加载提示

会话一开始注入了一个 `system-reminder`，声明很多工具并未直接加载，而是要先通过 `ToolSearch` 取 schema。

例如：

- `TaskCreate` / `TaskList` / `TaskUpdate`
- `WebFetch` / `WebSearch`
- `EnterPlanMode`
- 一些 Gmail / Google Calendar / Drive MCP 工具

这说明它的工具系统不是一次性把全部 schema 塞进上下文，而是：

1. 先告诉模型“有哪些工具名字可用”
2. 真要用时，再用 `ToolSearch select:...` 拉取精确 schema

这是比较典型的“工具延迟加载”设计。

### 2.3 Skills 提示

系统同时注入一组可用 skills：

- `update-config`
- `keybindings-help`
- `simplify`
- `fewer-permission-prompts`
- `loop`
- `claude-api`
- `init`
- `review`
- `security-review`

并且规则写得很硬：

- 用户显式输入 `/<skill-name>` 时必须调用 `Skill`
- 如果匹配请求，也应优先调用

但这次会话里，虽然 `Skill` 工具在工具列表中存在，实际上并没有真的被调用。也就是说，这次记录体现的是“skill 能力可见，但未成为主要工作流”。

### 2.4 项目级提示注入

系统把仓库里的 `CLAUDE.md` 全量注入到上下文中，内容包括：

- 项目介绍
- 架构规划
- 命令用法
- 关键 crate
- 网络环境
- 额外用户信息

这使 Claude Code 能在第一轮就带着项目背景工作，而不是完全靠临时探索。

### 2.5 自动记忆提示

还有一大段 auto memory 规范，定义了：

- memory 的类型：`user` / `feedback` / `project` / `reference`
- 什么时候该写 memory
- 什么时候不能写
- 如何写 frontmatter
- 如何维护 `MEMORY.md`

这说明 Claude Code 试图把“长期协作偏好”和“当前任务上下文”分离。

### 2.6 环境快照提示

系统还注入了环境与 git 快照：

- 当前目录
- 是否 git 仓库
- 分支
- 最近提交
- 启动时的 `git status`
- 模型版本、平台、shell

这类提示对代码代理很有价值，因为它减少了第一轮无效探测。

## 3. 它的 tool 使用方式

按最后一轮请求体中的真实 `tool_use` 统计，这次会话实际使用的是：

- `Bash`：17 次
- `Read`：13 次
- `Write`：11 次
- `Edit`：8 次
- `TaskCreate`：8 次
- `TaskUpdate`：7 次
- `ToolSearch`：1 次

没有实际调用：

- `Skill`
- `Agent`
- `WebSearch`
- `WebFetch`

### 3.1 典型工作流

它的真实工作流基本是：

1. `Bash` 做目录与文件存在性探测
2. `Read` 读取关键源码和配置
3. 先 `Write` 一份设计文档
4. 再通过 `Read` + `Edit` 多轮修订文档
5. 中途通过 `ToolSearch` 加载任务工具
6. 用 `TaskCreate` / `TaskUpdate` 把后续实现拆成若干任务

这是一种偏“工程执行型”的路径，而不是“重技能编排”或“多代理协作”路径。

### 3.2 ToolSearch 的作用

这次 `ToolSearch` 只做了一件事：加载 `TaskCreate`、`TaskUpdate`、`TaskList` 的 schema。

优点是上下文更省。
缺点是中途才加载，意味着：

- 需要额外一次工具调用
- 模型要先知道“哪些工具是 deferred 的”
- 提示词和工具系统耦合更深

### 3.3 Task 工具的使用特征

任务工具使用很重，说明 Claude Code 偏好：

- 把复杂工作拆成可追踪任务
- 显式维护“进行中/完成”状态
- 用任务列表替代纯文本计划

这对长会话、实现型任务是有帮助的，但对一次性的文档任务会显得偏重。

## 4. 它在 skill、tool、提示词上的优点

### 4.1 优点：提示词分层明确

它把“平台级行为规范”“项目级说明”“环境快照”“可用工具”“memory 规则”拆开注入，结构清晰。这样做的好处是：

- 每层职责明确
- 平台可以统一升级系统行为
- 项目只需维护 `CLAUDE.md`
- 工具与记忆能力可插拔

### 4.2 优点：工具体系工程化程度高

它区分了：

- 文件读写工具
- shell 工具
- 任务工具
- skill 工具
- agent 工具
- deferred tool loader

这种设计明显比“只有 shell”更适合审计、权限控制和 UI 展示。

### 4.3 优点：专用文件工具体验更稳定

从记录看，它虽然用了不少 `Bash`，但核心内容修改仍主要依赖 `Read` / `Write` / `Edit`。这带来几个好处：

- 变更粒度更清晰
- 不容易被 shell quoting 坑到
- UI 中更容易回溯“读了什么、改了什么”

### 4.4 优点：任务清单适合长流程

`TaskCreate` / `TaskUpdate` 让它能够把“设计 -> 模块拆分 -> 渲染器 -> writer -> 测试”这种多步实现过程结构化。对于复杂实现，确实比纯口头计划更稳。

### 4.5 优点：memory 规则比普通 agent 更成熟

它不仅支持 memory，还明确限制“哪些不该记”。这能减少把短期任务细节污染成长期记忆的问题。

### 4.6 优点：环境快照减少探测成本

像 git 分支、最近提交、工作目录、平台等都提前给出，能少做几轮无意义探测，对 agent 的首轮效率有帮助。

## 5. 它在 skill、tool、提示词上的缺点

### 5.1 缺点：提示词非常长，开销明显

从第一轮请求体可以看到，系统提示、skills、memory、环境说明都很长。好处是能力完整，坏处是：

- 首轮上下文成本高
- 每轮重复携带大量说明
- 容易把简单任务也包装成重流程

这次会话本质上只是“分析需求并写设计文档”，但系统层信息已经很厚了。

### 5.2 缺点：skill 在这次会话里基本只是“摆设”

系统明确暴露了很多 skill，但实际没有调用 `Skill`。这暴露出一个问题：

- 技能体系存在
- 但不一定真的成为主流程
- 用户很难从一次会话中感受到 skill 带来的收益

换句话说，这次记录里 skill 更像“可选扩展目录”，而不是高频执行机制。

### 5.3 缺点：Agent 能力也没有真正发挥

虽然 `Agent` 工具被定义得很完整，甚至有 `Explore`、`Plan` 等子代理类型，但这次会话没有实际使用。说明在中等复杂度任务中，Claude Code 仍然主要走单代理路径。

这带来的问题是：

- 提示里有很多关于 agent 的说明，但本次并未变成生产力
- 增加了系统复杂度
- 也增加了模型理解成本

### 5.4 缺点：Bash 依赖仍然偏重

尽管系统明确鼓励优先用专用工具，实际还是用了 17 次 `Bash`。这说明：

- 文件工具足够好，但目录探测、grep、存在性检查仍大量依赖 shell
- 一旦跨平台差异出现，shell 步骤的稳定性会下降
- 工具规范和真实行为之间还有落差

### 5.5 缺点：任务工具介入稍晚

它是先做了一轮设计文档修改，后面才 `ToolSearch` 加载任务工具并开始建任务。说明任务系统没有在一开始就统一工作流，而是“做到一半再切换成任务模式”。

这种方式的问题是：

- 前后工作流不一致
- 任务清单不是从头完整覆盖
- 任务状态和真实过程会有轻微错位

### 5.6 缺点：文档修改轮次偏多

它先 `Write` 一整份文档，再多轮 `Read` + `Edit` 局部修订。这种策略可行，但会带来：

- 内容抖动
- 重复读取同一文件
- 多次 patch 才收敛

如果前期先把命名规则、目录策略、冲突策略问清楚，再落盘，轮次可以更少。

## 6. 对 skill 的具体判断

如果只看这份记录，Claude Code 的 skill 机制有两个明显特点。

### 6.1 优点

- skill 的边界定义清楚
- 每个 skill 都写了触发条件和适用范围
- 很适合做“产品级固定工作流”，例如 review、security-review、claude-api

### 6.2 缺点

- 这次会话里没有实际调用，说明默认主路径不一定依赖 skill
- 如果用户不显式用 `/skill-name`，很多 skill 很可能只是挂在系统里
- 当 skill 没有真正进入主流程时，维护成本和心智负担会高于收益

结论是：它的 skill 设计本身不差，但从这次记录看，“可用”不等于“常用”。

## 7. 对 tool 的具体判断

如果把 tool 使用方式抽象一下，这次 Claude Code 的主打法是：

- 用 `Bash` 做外围探测
- 用 `Read/Write/Edit` 做核心文件操作
- 用 `Task*` 做流程管理
- 用 `ToolSearch` 做延迟 schema 获取

这是一个比较稳的组合，但也说明它更偏“受控 IDE 工具链”，不是“极简 shell agent”。

### 7.1 适合它的场景

- 有明确仓库上下文
- 需要长期追踪任务状态
- 需要严格区分读取、修改、执行
- 需要做工具权限管理或审计

### 7.2 不太适合它的场景

- 非常轻量的一次性问题
- 只需少量上下文就能回答的问答
- 不值得引入 task/skill/agent 多层机制的简单工作

## 8. 总结

基于这份记录，Claude Code 的核心风格可以概括为：

- 提示词上：分层、重规则、重环境注入
- 工具上：专用文件工具 + shell + task 的混合模式
- skill 上：体系完整，但本次未真正成为主工作流
- agent 上：能力暴露充分，但本次未实际启用

它的主要优点是工程化、可控、可审计、适合中长任务。
它的主要缺点是上下文偏重、系统复杂度高、skill/agent 在实际会话中未必真正带来收益。

如果把这份记录当成设计参考，一个很值得借鉴的点是：

- 把“项目说明、环境快照、工具能力、长期记忆规则”显式分层

一个很值得避免的点是：

- 不要把太多低频机制都塞进默认提示里，否则简单任务也会被重型工作流拖慢。
