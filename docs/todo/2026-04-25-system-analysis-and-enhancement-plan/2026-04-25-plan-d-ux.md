# Design Doc: Plan D - UX Deep Dive (用户体验深度增强)

**Date**: 2026-04-25
**Status**: Draft
**Author**: Claude Code

## 1. Current State (现状)

目前的 Zero-Nova 桌面端应用已经实现了基础的 AI 对话功能，但在交互细节和视觉反馈上仍处于“工具化”阶段，缺乏“智能化”和“沉浸式”的体验。

### 1.1 工具执行过程的“黑盒”感
当 Agent 调用工具（如运行 Bash、搜索网页）时，用户只能看到一个简单的状态指示器（如“正在执行...”）。
- **问题**: 用户无法预知工具执行的进度或细节，这种不确定性会导致用户产生“程序卡死”的错觉，尤其是在执行耗时较长的任务时。

### 1.2 思考过程的视觉层级较低
虽然系统支持 `ThinkingDelta` 的流式推送，但前端在展示“思考”与“最终输出”时，视觉区分度不够。
- **问题**: 思考过程与对话内容在视觉上混杂在一起，用户难以一眼分辨哪些是模型的推理逻辑，哪些是最终的指令或回答。

### 1.3 交互反馈的碎片化
对于重要的系统事件（如工具权限请求、长任务完成、Agent 切换成功），目前的反馈方式较为单一。
- **问题**: 缺乏一种贯穿全局、既不打扰用户又足够醒目的交互语言。

---

## 2. Goals (目标)

1.  **实现工具执行的“透明化”**: 让用户能够实时、直观地看到工具运行的中间过程（如 Terminal 输出）。
2.  **优化思考过程的视觉表达**: 建立清晰的视觉层级，将“思考”与“回答”彻底解耦。
3.  **打造沉浸式交互体验**: 通过更丰富的 UI 组件（如进度条、动态图标、状态卡片）提升整体产品的精致感。

---

## 3. Detailed Design (实现细节)

### 3.1 工具执行的可视化 (Tool Execution Transparency)

#### 3.1.1 实时输出捕获 (Stdout Capture)
- **后端改进**: 增强 `ToolRegistry` 的实现，使其不仅返回执行结果，还能实时通过事件流推送工具的标准输出（stdout）。
- **协议扩展**: 在 `nova-protocol` 中增加 `ToolOutputDelta` 消息类型。
```rust
pub enum MessageEnvelope {
    // ...
    ToolOutputDelta(ToolOutputPayload),
}

pub struct ToolOutputPayload {
    pub tool_id: String,
    pub delta: String, // 实时输出的片段
}
```

#### 3.1.2 前端“迷你终端”组件
- **实现**: 在聊天窗口中为正在运行的工具创建一个“迷你终端”卡片。
- **交互**: 该卡片会随着 `ToolOutputDelta` 的到达而动态滚动内容，类似于真实的 Terminal 体验。

### 3.2 思考过程的视觉重构 (Thinking Process Refinement)

#### 3.2.1 视觉分层设计
不再将思考内容直接混入对话流，而是采用独立的“思考容器”。
- **设计元素**:
    - **折叠设计**: 默认情况下，思考过程以一个带有“思考中...”动画的折叠面板显示，点击可展开查看详情。
    - **样式区分**: 使用不同的背景色（如浅灰色或淡蓝色）、字体（等宽字体）和图标（如大脑或灯泡图标）来标识思考区块。
- **动效**: 使用微小的脉冲动画或流光效果来表示 Agent 正在“深度思考”中。

### 3.3 全局交互语言 (Global Interaction Language)

#### 3.3.1 任务进度感 (Sense of Progress)
- **全局进度条**: 对于跨会话的长任务或大文件处理，在应用顶部或侧边栏显示一个全局进度指示器。
- **状态卡片**: 当 Agent 完成一个复杂的工具链调用时，弹出一个精致的“任务完成”卡片，总结执行结果。

#### 3.3.2 权限请求流 (Permission Flow)
- **拦截式交互**: 当 Agent 需要执行高危工具（如 `rm -rf`）时，前端弹出一个带有“允许/拒绝”选项的半透明 Modal，而不是简单的系统提示，以此增强安全感。

## 4. Test Plan (测试计划)

### 4.1 视觉与动效测试
- **视觉层级验证**: 邀请非开发人员测试，观察其是否能一眼区分“思考”与“回答”。
- **动画性能**: 在低端配置的机器上测试“迷你终端”滚动和思考过程动画是否平滑。

### 4.2 功能性测试
- **输出完整性**: 运行复杂的 Shell 脚本，验证“迷你终端”捕获的输出是否完整、无乱码且顺序正确。
- **权限流程闭环**: 验证从触发高危工具 $\rightarrow$ 弹出权限请求 $\rightarrow$ 用户点击 $\rightarrow$ 后端继续执行的整个链路。

### 4.3 用户体验评估 (UX Survey)
- 通过对比测试（改进前 vs 改进后），评估用户在处理复杂任务时的焦虑感是否降低，以及对系统响应速度的感知。
